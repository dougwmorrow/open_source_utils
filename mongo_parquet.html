üîß How MongoDB Compression Works

Storage Engine: WiredTiger (MongoDB's default storage engine since v3.2)

Compression Timing: Automatic during data insertion - no manual intervention required

Compression Algorithms Available:

  • Snappy (default): 1.5-2x compression, minimal CPU overhead (~5%)
  • Zstd (recommended for archival): 2-2.5x compression, moderate CPU overhead (5-15%)
  • Zlib: 2-2.8x compression, higher CPU overhead (10-20%)

Configuration: Set at collection level during creation or migration

db.createCollection("timeseries", {
    storageEngine: {
        wiredTiger: {
            configString: "block_compressor=zstd"
        }
    }
})

MongoDB's BSON format creates a 24-48 hour migration bottleneck to cloud platforms

Critical Limitation
MongoDB stores ALL data in BSON format‚Äîit cannot store or serve Parquet files as columnar data. Even if you use GridFS to store Parquet files, they are converted to binary chunks and lose all columnar query benefits.

Every major cloud platform expects Parquet as the standard format

  • Snowflake, Databricks, BigQuery: Native Parquet support
  • AWS Athena: Cannot query MongoDB directly
  • Azure Synapse: Requires expensive transformation pipelines

MongoDB is NOT an object storage service

  • MongoDB is a database that stores documents in BSON format
  • GridFS splits files into 255KB chunks stored as BSON documents
  • Cannot act as S3-compatible storage or serve files in their original format
  • All data must be extracted and transformed for analytical platforms

Migration complexity differs dramatically between formats

MongoDB Migration Process

  1. Export from MongoDB (12-24 hours)
  2. Transform BSON to Parquet (6-12 hours)
  3. Upload to cloud platform (6-12 hours)
  4. Total: 24-48 hours + complexity

Parquet Migration Process

  1. Upload files to cloud (2-6 hours)
  2. Register in catalog (minutes)
  3. Ready to query
  4. Total: 2-6 hours, simple copy
Migration Speed Impact
Parquet migrates 8-10x faster with zero transformation risk. This difference compounds with larger datasets.

BSON stores data as binary-encoded documents, requiring full document reads for any field access

How BSON (Binary JSON) Storage Works

BSON Document Structure

Original JSON Document

{
  "_id": "507f1f77bcf86cd799439011",
  "timestamp": "2024-01-15T10:30:00Z",
  "sensor_id": "TEMP_001",
  "temperature": 72.5,
  "humidity": 45,
  "pressure": 1013.25,
  "location": {
    "building": "A",
    "floor": 3,
    "room": "301"
  }
}

BSON Binary Representation

/* Header: Document Size */
00 00 00 87
/* Field: _id (ObjectId) */
07 5F 69 64 00 50 7F 1F 77...
/* Field: timestamp (String) */
02 74 69 6D 65 73 74 61 6D...
/* Field: temperature (Double) */
01 74 65 6D 70 65 72 61 74...
/* Nested doc: location */
03 6C 6F 63 61 74 69 6F 6E...

Key Characteristics:

  • Sequential Storage: All fields stored together in each document
  • Type Information: Each field includes type markers (01=double, 02=string, etc.)
  • Self-Contained: Each document is independent with all its data
  • Variable Length: Documents can have different sizes and fields
  • Read Pattern: Must read entire document to access any field
GridFS Storage Clarification

Even when storing Parquet files in MongoDB GridFS:

  1. Files are split into 255KB chunks
  2. Each chunk is stored as BSON Binary data
  3. Files must be reassembled before use
  4. Columnar benefits are completely lost
  5. Cannot query data in Parquet format

Result: GridFS is file storage within MongoDB, NOT object storage like S3

Advanced Compression Techniques:

  • Dictionary Encoding: Repeated values stored once (e.g., sensor locations)
  • Run-Length Encoding: Sequential identical values compressed

  • Page-Level Compression: Snappy/Gzip on top of encoding
Query Optimization Features
  • Predicate Pushdown: Skip entire row groups using min/max statistics
  • Column Pruning: Read only requested columns, ignore others
  • Late Materialization: Filter before reconstructing rows
  • Vectorized Processing: Process data in CPU-efficient batches

BSON/MongoDB Query Execution

Step 1: Full Collection Scan

// MongoDB must read every document
db.timeseries.find({
  timestamp: {
    $gte: "2024-01-01", 
    $lt: "2024-02-01"
  },
  "location.building": "A"
}).forEach(doc => 
  temps.push(doc.temperature)
)

Data Read: 170GB (compressed)
Documents Scanned: 4 billion
Useful Data: ~1.5GB

Performance Bottlenecks

  • Decompress each document block
  • Parse BSON structure
  • Navigate nested location object
  • 97% of data read is discarded
  • No parallel column processing

Parquet Query Execution

Step 1: Smart Data Access

# PyArrow reads only needed columns
df = pq.read_table('data.parquet',
    columns=['timestamp', 
             'temperature', 
             'location'],
    filters=[
        ('timestamp', '>=', '2024-01-01'),
        ('timestamp', '<', '2024-02-01'),
        ('location', '==', 'A')
    ])

Data Read: 3.5GB
Row Groups Skipped: 87%
Useful Data: ~1.5GB

Performance Optimizations

  • Row group statistics filter early
  • Read only 3 columns of 20+
  • Dictionary encoding for location
  • Vectorized aggregation
  • Parallel column decompression
